Foreign Field Addition RFC

This document is meant to explain the foreign field addition gate in Kimchi.

Overview​

The goal of this gate is to perform the following operation between a left aaa and a right bbb input, to obtain a result rrr a+s⋅b=rmod  fa + s \cdot b = r \mod fa+s⋅b=rmodf where a,b,r∈Ffa,b,r\in\mathbb{F}_fa,b,r∈Ff​ belong to the foreign field Ff\mathbb{F}_fFf​ of modulus fff and we work over a native field Fn\mathbb{F}_nFn​ of modulus nnn, and s∈{−1,1}s \in \{-1, 1\}s∈{−1,1} is a flag indicating whether it is a subtraction or addition gate.

If f<2⋅nf < 2 \cdot nf<2⋅n then we can easily perform the above computation natively since no overflows happen. But in this gate we are interested in the contrary case in which f>nf>nf>n. In order to deal with this, we will divide foreign field elements into limbs that fit in our native field. We want to be compatible with the foreign field multiplication gate, and thus the parameters we will be using are the following:

  • 3 limbs of 88 bits each (for a total of 264 bits)
  • So f<2264f < 2^{264}f<2264, concretely:
    • The modulus fff of the foreign field Ff\mathbb{F}_fFf​ is 256 bit
    • The modulus of our the native field Fn\mathbb{F}_nFn​ is 255 bit

In other words, using 3 limbs of 88 bits each allows us to represent any foreign field element in the range [0,2264)[0,2^{264})[0,2264) for foreign field addition, but only up to 22592^{259}2259 for foreign field multiplication. Thus, with the current configuration of our limbs, our foreign field must be smaller than 22592^{259}2259 (because 2264⋅2255>22592+22592^{264} \cdot 2^{255} > {2^{259}}^2 + 2^{259}2264⋅2255>22592+2259, more on this in Foreign Field Multiplication or the original FFmul RFC.

We will perform the addition in 3 steps, one per limb. Now, when we split long additions in this way, we must be careful with carry bits between limbs. Also, even if aaa and bbb are foreign field elements, it could be the case that a+ba + ba+b is already larger than the modulus (in this case a+ba + ba+b could be at most 2f−22f - 22f−2). First, if a+ba + ba+b is larger than fff, then we will have a field overflow (represented by q=1q = 1q=1) and thus will have to subtract fff from the sum a+ba + ba+b to obtain rrr. Whereas the foreign field overflow necessitates an overflow bit qqq for the foreign field equation above, when q=1q = 1q=1 there is a corresponding subtraction that may introduce carries (or even borrows) between the limbs. This is because r=a+b−q⋅fmod  2264r = a + b - q \cdot f \mod 2^{264}r=a+b−q⋅fmod2264. Therefore, in the equations for the limbs we will use a carry flag cic_ici​ for limb iii to represent both carries and borrows. The carry flags cic_ici​ are in {−1,0,1}\{-1, 0, 1\}{−1,0,1}, where ci=−1c_i = -1ci​=−1 represents a borrow and ci=1c_i = 1ci​=1 represents a carry. Next we explain how this works.

In order to perform this operation in parts, we first take a look at the least significant limb, which is the easiest part. This means we want to know how to compute r0r_0r0​. First, if the addition of the bits in a0a_0a0​ and b0b_0b0​ produce a carry (or borrow) bit, then it should propagate to the second limb. That means one has to subtract 2882^{88}288 from a0+b0a_0 + b_0a0​+b0​, add 111 to a1+b1a_1 + b_1a1​+b1​ and set the low carry flag c0c_0c0​ to 1 (otherwise it is zero). a0+s⋅b0=q⋅f0+r0+c0⋅288a_0 + s \cdot b_0 = q \cdot f_0 + r_0 + c_0 \cdot 2^{88}a0​+s⋅b0​=q⋅f0​+r0​+c0​⋅288

This kind of equation needs an additional check that the carry coefficient c0c_0c0​ is a correct carry value (belonging to {−1,0,1}\{-1,0,1\}{−1,0,1}). We will use this idea for the remaining limbs as well.

Looking at the second limb, we first need to observe that the addition of a1a_1a1​ and b1b_1b1​ can, not only produce a carry bit c1c_1c1​, but they may need to take into account the carry bit from the first limb; c0c_0c0​. Similarly to the above,

Note that in this case, the carry coefficient from the least significant limb is not multiplied by 2882^{88}288, but instead is used as is, since it occupies the least significant position of the second limb. Here, the second carry bit c1c_1c1​ is the one being followed by 888888 zeros. Again, we will have to check that c1c_1c1​ is a bit.

Finally, for the third limb, we obtain a similar equation. But in this case, we do not have to take into account c0c_0c0​ anymore, since it was already considered within c1c_1c1​. Here, the most significant carry bit c2c_2c2​ should always be a zero so we can ignore it.

Our witness computation is currently using the BigUint library, which takes care of all of the intermediate carry limbs itself so we do not have to address all casuistries ourselves (such as propagating the low carry flag to the high limb in case the middle limb is all zeros).

Upper bound check for foreign field membership​

Last but not least, we should perform some range checks to make sure that the result rrr is contained in Ff\mathbb{F}_fFf​. This is important because there could be other values of the result which still fit in <2264<2^{264}<2264 but are larger than fff, and we must make sure that the final result is the minimum one (that we will be referring to as rminr_{min}rmin​ in the following).

Ideally, we would like to reuse some gates that we already have. In particular, we can perform range checks for 0≤X<23ℓ=23⋅880\leq X <2^{3\ell}=2^{3\cdot 88}0≤X<23ℓ=23⋅88. Thus, we can perform the standard rmin∈[0,2264)r_{min} \in [0,2^{264})rmin​∈[0,2264) check together with the rmin−f+2264∈[0,2264)r_{min} - f + 2^{264} \in [0,2^{264})rmin​−f+2264∈[0,2264), which together will imply rmin∈[0,f)r_{min} \in [0,f)rmin​∈[0,f) as intended. Subtractions​

Mathematically speaking, a subtraction within a field is no more than an addition over that field. Negative elements are not different from "positive" elements in finite fields (or in any modular arithmetic). Our witness computation code computes negative sums by adding the modulus to the result. To give a general example, the element −e-e−e within a field Fm\mathbb{F}_mFm​ of order mmm and e<me < me<m is nothing but m−em - em−e. Nonetheless, for arbitrarily sized elements (not just those smaller than the modulus), the actual field element could be any c⋅m−ec \cdot m - ec⋅m−e, for any multiple c⋅mc \cdot mc⋅m of the modulus. Thus, representing negative elements directly as "absolute" field elements may incur in additional computations involving multiplications and thus would result in a less efficient mechanism.

Instead, our gate encodes subtractions and additions directly within the sign term that is multiplying the right input. This way, there is no need to check that the negated value is performed correctly (which would require an additional row for a potential FFNeg gate).

Gadget​

The full foreign field addition/subtraction gadget will be composed of:

  • 111 public input row containing the value 111;
  • nnn rows with ForeignFieldAdd gate type:
    • for the actual nnn chained additions or subtractions;
  • 111 ForeignFieldAdd row for the bound addition;
  • 111 Zero row for the final bound addition.
  • 111 RangeCheck gadget for the first left input of the chain a1:=aa_1 := aa1​:=a;
  • Then, nnn of the following set of rows:
    • 111 RangeCheck gadget for the iii-th right input bib_ibi​;
    • 111 RangeCheck gadget for the iii-th result which will correspond to the (i+1)(i+1)(i+1)-th left input of the chain ri=ai+1r_i = a_{i+1}ri​=ai+1​.
  • 111 final RangeCheck gadget for the bound check uuu.

This mechanism can chain foreign field additions together. Initially, there are nnn foreign field addition gates, followed by a foreign field addition gate for the bound addition (whose current row corresponds to the next row of the last foreign field addition gate), and an auxiliary Zero row that holds the upper bound. At the end, an initial left input range check is performed, which is followed by a nnn pairs of range check gates for the right input and intermediate result (which become the left input for the next iteration). After the chained inputs checks, a final range check on the bound takes place.

Optimizations​

When we use this gate as part of a larger chained gadget, we should optimize the number of range check rows to avoid redundant checks. In particular, if the result of an addition becomes one input of another addition, there is no need to check twice that the limbs of that term have the right length.

The sign is now part of the coefficients of the gate. This will allow us to remove the sign check constraint and release one witness cell element. But more importantly, it brings soundness to the gate as it is now possible to wire the 111 public input to the overflow witness of the final bound check of every addition chain.
