Plookup\plookupPlookup in Kimchi

In 2020, Plookup\plookupPlookup showed how to create lookup proofs. Proofs that some witness values are part of a lookup table. Two years later, an independent team published plonkup showing how to integrate Plookup\plookupPlookup into PlonK\plonkPlonK.

This document specifies how we integrate Plookup\plookupPlookup in kimchi. It assumes that the reader understands the basics behind Plookup\plookupPlookup.

Overview​

We integrate Plookup\plookupPlookup in kimchi with the following differences:

  • we snake-ify the sorted table instead of wrapping it around (see later)
  • we allow fixed-ahead-of-time linear combinations of columns of the queries we make
  • we implemented different tables, like RangeCheck and XOR.
  • we allow several lookups (or queries) to be performed within the same row
  • zero-knowledgeness is added in a specific way (see later)

The following document explains the protocol in more detail

Recap on the grand product argument of Plookup\plookupPlookup​

As per the Plookup\plookupPlookup paper, the prover will have to compute three vectors:

  • fff, the (secret) query vector, containing the witness values that the prover wants to prove are part of the lookup table.
  • ttt, the (public) lookup table.
  • sss, the (secret) concatenation of fff and ttt, sorted by ttt (where elements are listed in the order they are listed in ttt).

Essentially, Plookup\plookupPlookup proves that all the elements in fff are indeed in the lookup table ttt if and only if the following multisets are equal:

  • {(1+β)f,diff(t)}\{(1+\beta)f, \text{diff}(t)\}{(1+β)f,diff(t)}
  • diff(sorted(f,t))\text{diff}(\text{sorted}(f, t))diff(sorted(f,t))

where diff\text{diff}diff is a new set derived by applying a "randomized difference" between every successive pairs of a vector, and (f,t)(f, t)(f,t) is the set union of fff et ttt.

More precisely, for a set S={s0,s1,⋯ ,sn}S = \{s_{0}, s_{1}, \cdots, s_{n} \}S={s0​,s1​,⋯,sn​}, diff(S)\text{diff}(S)diff(S) is defined as the set {s0+βs1,s1+βs2,⋯ ,sn−1+βsn}\{s_{0} + \beta s_{1}, s_{1} + \beta s_{2}, \cdots, s_{n - 1} + \beta s_{n}\}{s0​+βs1​,s1​+βs2​,⋯,sn−1​+βsn​}.

For example, with:

  • f={5,4,1,5}f = \{5, 4, 1, 5\}f={5,4,1,5}
  • t={1,4,5}t = \{1, 4, 5\}t={1,4,5}

we have:

  • sorted(f,t)={1,1,4,4,5,5,5}\text{sorted}(f,t) = \{1, 1, 4, 4, 5, 5, 5\}sorted(f,t)={1,1,4,4,5,5,5}
  • {(1+β)f,diff(t)}={(1+β)5,(1+β)4,(1+β)1,(1+β)5,1+β4,4+β5}\{\color{blue}{(1+\beta)f}, \color{green}{\text{diff}(t)}\} = \{\color{blue}{(1+\beta)5, (1+\beta)4, (1+\beta)1, (1+\beta)5}, \color{green}{1+\beta 4, 4+\beta 5}\}{(1+β)f,diff(t)}={(1+β)5,(1+β)4,(1+β)1,(1+β)5,1+β4,4+β5}
  • diff(sorted(f,t))={1+β1,1+β4,4+β4,4+β5,5+β5,5+β5}\text{diff}(\text{sorted}(f, t)) = \{1+\beta 1, 1+\beta 4, 4+\beta 4, 4+\beta 5, 5+\beta 5, 5+\beta 5\}diff(sorted(f,t))={1+β1,1+β4,4+β4,4+β5,5+β5,5+β5}

Note: This assumes that the lookup table is a single column. You will see in the next section how to address lookup tables with more than one column.

Querying the table​

The Plookup\plookupPlookup paper handles a vector of lookups fff which we do not have. So the first step is to create such a table from the witness columns (or registers). To do this, we define the following objects:

  • a query tells us what registers, in what order, and scaled by how much, are part of a query
  • a query selector tells us which rows are using the query. It is pretty much the same as a gate selector.

Let's go over the first item in this section.

For example, the following query tells us that we want to check if r0⊕r2=2⋅r1r_0 \oplus r_2 = 2\cdot r_1r0​⊕r2​=2⋅r1​

lro
1, r0r_0r0​1, r2r_2r2​2, r1r_1r1​

r0r_0r0​, r1r_1r1​ and r2r_2r2​ will be the result of the evaluation at gig^igi of respectively the wire polynomials w0w_0w0​, w1w_1w1​ and w2w_2w2​. To perform vector lookups (i.e. lookups over a list of values, not a single element), we use a standard technique which consists of coining a combiner value jjj and sum the individual elements of the list using powers of this coin.

The grand product argument for the lookup constraint will look like this at this point:

acci=acci−1⋅(1+β)⋅(γ+j0⋅1⋅w0(gi)+j⋅1⋅w2(gi)+j2⋅2⋅w1(gi))⋅(γ(1+β)+ti−1+βti)(γ(1+β)+si−1+βsi)(γ(1+β)+sn+i−1+βsn+i)\mathsf{acc}_i = \mathsf{acc}_{i-1} \cdot \frac{(1+\beta) \cdot {\color{green}(\gamma + j^0 \cdot 1 \cdot w_0(g^i) + j \cdot 1 \cdot w_2(g^i) + j^2 \cdot 2 \cdot w_1(g^i))} \cdot (\gamma(1 + \beta) + t_{i-1} + \beta t_i)}{(\gamma(1+\beta) + s_{i-1} + \beta s_{i})(\gamma(1+\beta) + s_{n+i-1} + \beta s_{n+i})}acci​=acci−1​⋅(γ(1+β)+si−1​+βsi​)(γ(1+β)+sn+i−1​+βsn+i​)(1+β)⋅(γ+j0⋅1⋅w0​(gi)+j⋅1⋅w2​(gi)+j2⋅2⋅w1​(gi))⋅(γ(1+β)+ti−1​+βti​)​

Not all rows need to perform queries into a lookup table. We will use a query selector in the next section to make the constraints work with this in mind.

Supporting multiple queries​

Since we would like to allow multiple table lookups per row, we define multiple queries, where each query is associated with a lookup selector.

Previously, ChaCha20 was implemented in Kimchi but has been removed as it has become unneeded. You can still find the implementation here. The ChaCha gates all perform 444 queries in a row. Thus, 444 is trivially the largest number of queries that happen in a row.

Important: to make constraints work, this means that each row must make 444 queries. Potentially some or all of them are dummy queries.

For example, the ChaCha0, ChaCha1, and ChaCha2 gates will jointly apply the following 4 XOR queries on the current and following rows:

lro-lro-lro-lro
1, r3r_3r3​1, r7r_7r7​1, r11r_{11}r11​-1, r4r_4r4​1, r8r_8r8​1, r12r_{12}r12​-1, r5r_5r5​1, r9r_9r9​1, r13r_{13}r13​-1, r6r_6r6​1, r10r_{10}r10​1, r14r_{14}r14​

which you can understand as checking for the current and following row that

r3⊕r7=r11r4⊕r8=r12r5⊕r9=r13r6⊕r10=r14\begin{aligned} r_3 \oplus r_7 &= r_{11}\\ r_4 \oplus r_8 &= r_{12}\\ r_5 \oplus r_9 &= r_{13}\\ r_6 \oplus r_{10} &= r_{14} \end{aligned}r3​⊕r7​r4​⊕r8​r5​⊕r9​r6​⊕r10​​=r11​=r12​=r13​=r14​​

The ChaChaFinal also performs 444 (somewhat similar) queries in the XOR lookup table. In total this is 888 different queries that could be associated to 888 selector polynomials.

Grouping queries by queries pattern​

Associating each query with a selector polynomial is not necessarily efficient. To summarize:

  • the ChaCha0, ChaCha1, and ChaCha2 gates that in total make 444 queries into the XOR table
  • the ChaChaFinal gate makes another 444 different queries into the XOR table

Using the previous section's method, we'd have to use 888 different lookup selector polynomials for each of the different 888 queries. Since there's only 222 use-cases, we can simply group them by queries patterns to reduce the number of lookup selector polynomials to 222.

The grand product argument for the lookup constraint looks like this now:

acci=acci−1⋅(1+β)4⋅query⋅(γ(1+β)+ti−1+βti)(γ(1+β)+si−1+βsi)×…\mathsf{acc}_i = \mathsf{acc}_{i-1} \cdot \frac{{\color{green}(1+\beta)^4 \cdot \mathsf{query}} \cdot (\gamma(1 + \beta) + t_{i-1} + \beta t_i)}{(\gamma(1+\beta) + s_{i-1} + \beta s_{i})\times \ldots}acci​=acci−1​⋅(γ(1+β)+si−1​+βsi​)×…(1+β)4⋅query⋅(γ(1+β)+ti−1​+βti​)​

where query\color{green}{\mathsf{query}}query is constructed as:

query= selector1⋅pattern1+ selector2⋅pattern2+ (1−selector1−selector2)⋅(γ+0+j⋅0+j2⋅0)4\begin{aligned} \mathsf{query} = &\ \mathsf{selector}_1 \cdot \mathsf{pattern}_1 + \\ &\ \mathsf{selector}_2 \cdot \mathsf{pattern}_2 + \\ &\ (1 - \mathsf{selector}_1 - \mathsf{selector}_2) \cdot (\gamma + 0 + j \cdot 0 + j^2 \cdot 0)^4 \end{aligned}query=​ selector1​⋅pattern1​+ selector2​⋅pattern2​+ (1−selector1​−selector2​)⋅(γ+0+j⋅0+j2⋅0)4​

where, for example the first pattern for the ChaCha0, ChaCha1, and ChaCha2 gates looks like this:

pattern1= (γ+w3(gi)+j⋅w7(gi)+j2⋅w11(gi))⋅ (γ+w4(gi)+j⋅w8(gi)+j2⋅w12(gi))⋅ (γ+w5(gi)+j⋅w9(gi)+j2⋅w13(gi))⋅ (γ+w6(gi)+j⋅w10(gi)+j2⋅w14(gi))⋅\begin{aligned} \mathsf{pattern}_1 = &\ (\gamma + w_3(g^i) + j \cdot w_7(g^i) + j^2 \cdot w_{11}(g^i)) \cdot \\ &\ (\gamma + w_4(g^i) + j \cdot w_8(g^i) + j^2 \cdot w_{12}(g^i)) \cdot \\ &\ (\gamma + w_5(g^i) + j \cdot w_9(g^i) + j^2 \cdot w_{13}(g^i)) \cdot \\ &\ (\gamma + w_6(g^i) + j \cdot w_{10}(g^i) + j^2 \cdot w_{14}(g^i)) \cdot \\ \end{aligned}pattern1​=​ (γ+w3​(gi)+j⋅w7​(gi)+j2⋅w11​(gi))⋅ (γ+w4​(gi)+j⋅w8​(gi)+j2⋅w12​(gi))⋅ (γ+w5​(gi)+j⋅w9​(gi)+j2⋅w13​(gi))⋅ (γ+w6​(gi)+j⋅w10​(gi)+j2⋅w14​(gi))⋅​

Note that there's now 444 dummy queries, and they only appear when none of the lookup selectors are active. If a pattern uses less than 444 queries, it has to be padded with dummy queries as well.

Finally, note that the denominator of the grand product argument is incomplete in the formula above. Since the nominator has degree 555 in γ(1+β)\gamma (1 + \beta)γ(1+β), the denominator must match too. This is achieved by having a longer sss, and referring to it 555 times. The denominator thus becomes ∏k=15(γ(1+β)+skn+i−1+βskn+i)\prod_{k=1}^{5} (\gamma (1+\beta) + s_{kn+i-1} + \beta s_{kn+i})∏k=15​(γ(1+β)+skn+i−1​+βskn+i​).

Back to the grand product argument​

There are two things that we haven't touched on:

  • The vector ttt representing the combined lookup table (after its columns have been combined with a joint combiner jjj). The non-combined lookup table is fixed at setup time and derived based on the lookup tables used in the circuit.
  • The vector sss representing the sorted multiset of both the queries and the lookup table. This is created by the prover and sent as commitment to the verifier.

The first vector ttt is quite straightforward to think about:

  • if it is smaller than the domain (of size nnn), then we can repeat the last entry enough times to make the table of size nnn.
  • if it is larger than the domain, then we can either increase the domain or split the vector in two (or more) vectors. This is most likely what we will have to do to support multiple lookup tables later.

What about the second vector sss?

The sorted vector sss​

We said earlier that in original PlonK\plonkPlonK the size of sss is equal to ∣s∣=∣f∣+∣t∣|s| = |f|+|t|∣s∣=∣f∣+∣t∣, where fff encodes queries, and ttt encodes the lookup table. With our multi-query approach, the second vector sss is of the size

n⋅∣#queries∣+∣lookup_table∣n \cdot |\#\text{queries}| + |\text{lookup\_table}|n⋅∣#queries∣+∣lookup_table∣

That is, it contains the nnn elements of each query vectors (the actual values being looked up, after being combined with the joint combinator, that's 444 per row), as well as the elements of our lookup table (after being combined as well).

Because the vector sss is larger than the domain size nnn, it is split into several vectors of size nnn. Specifically, in the plonkup paper, the two halves of sss, which are then interpolated as h1h_1h1​ and h2h_2h2​. The denominator in PlonK\plonkPlonK in the vector form is

(γ(1+β)+si−1+βsi)(γ(1+β)+sn+i−1+βsn+i)\big(\gamma(1+\beta) + s_{i-1} + \beta s_{i}\big)\big(\gamma(1+\beta)+s_{n+i-1} + \beta s_{n+i}\big)(γ(1+β)+si−1​+βsi​)(γ(1+β)+sn+i−1​+βsn+i​)

which, when interpolated into h1h_1h1​ and h2h_2h2​, becomes

(γ(1+β)+h1(x)+βh1(g⋅x))(γ(1+β)+h2(x)+βh2(g⋅x))\big(\gamma(1+\beta) + h_1(x) + \beta h_1(g \cdot x)\big)\big(\gamma(1+\beta) + h_2(x) + \beta h_2(g \cdot x)\big)(γ(1+β)+h1​(x)+βh1​(g⋅x))(γ(1+β)+h2​(x)+βh2​(g⋅x))

Since one has to compute the difference of every contiguous pairs, the last element of the first half is the replicated as the first element of the second half (sn−1=sns_{n-1} = s_{n}sn−1​=sn​). Hence, a separate constraint must be added to enforce that continuity on the interpolated polynomials h1h_1h1​ and h2h_2h2​:

Ln−1(X)⋅(h1(X)−h2(g⋅X))≡0L_{n-1}(X)\cdot\big(h_1(X) - h_2(g \cdot X)\big) \equiv 0Ln−1​(X)⋅(h1​(X)−h2​(g⋅X))≡0

which is equivalent to checking that h1(gn−1)=h2(1)h_1(g^{n-1}) = h_2(1)h1​(gn−1)=h2​(1).

The sorted vector sss in kimchi​

Since this vector is known only by the prover, and is evaluated as part of the protocol, zero-knowledge must be added to the corresponding polynomial (in case of Plookup\plookupPlookup approach, to h1(X),h2(X)h_1(X),h_2(X)h1​(X),h2​(X)). To do this in kimchi, we use the same technique as with the other prover polynomials: we randomize the last evaluations (or rows, on the domain) of the polynomial.

This means two things for the lookup grand product argument:

  1. We cannot use the wrap around trick to make sure that the list is split in two correctly (enforced by Ln−1(h1(x)−h2(g⋅x))=0L_{n-1}(h_1(x) - h_2(g \cdot x)) = 0Ln−1​(h1​(x)−h2​(g⋅x))=0 which is equivalent to h1(gn−1)=h2(1)h_1(g^{n-1}) = h_2(1)h1​(gn−1)=h2​(1) in the Plookup\plookupPlookup paper)
  2. We have even less space to store an entire query vector. Which is actually super correct, as the witness also has some zero-knowledge rows at the end that should not be part of the queries anyway.

The first problem can be solved in two ways:

  • Zig-zag technique. By reorganizing sss to alternate its values between the columns. For example, h1=(s0,s2,s4,⋯ )h_1 = (s_0, s_2, s_4, \cdots)h1​=(s0​,s2​,s4​,⋯) and h2=(s1,s3,s5,⋯ )h_2 = (s_1, s_3, s_5, \cdots)h2​=(s1​,s3​,s5​,⋯) so that you can simply write the denominator of the grand product argument as (γ(1+β)+h1(x)+βh2(x))(γ(1+β)+h2(x)+βh1(x⋅g))(\gamma(1+\beta) + h_1(x) + \beta h_2(x))(\gamma(1+\beta)+ h_2(x) + \beta h_1(x \cdot g))(γ(1+β)+h1​(x)+βh2​(x))(γ(1+β)+h2​(x)+βh1​(x⋅g)) Whis approach is taken by the plonkup paper.
  • Snake technique. By reorganizing sss as a snake. This is what is currently implemented in kimchi.

The snake technique rearranges sss into the following shape:

                           __    _
          s_0 |  s_{2n-1} |  |  | |

(γ(1+β)+h1(x)+βh1(x⋅g))(γ(1+β)+h2(x⋅g)+βh2(x))\big(\gamma(1+\beta) + h_1(x) + \beta h_1(x \cdot g)\big)\big(\gamma(1+\beta)+ h_2(x \cdot g) + \beta h_2(x)\big)(γ(1+β)+h1​(x)+βh1​(x⋅g))(γ(1+β)+h2​(x⋅g)+βh2​(x))

Unsorted ttt in sss​

Note that at setup time, ttt cannot be sorted lexicographically as it is not combined yet. Since sss must be sorted by ttt (in other words sorting of sss must follow the elements of ttt), there are two solutions:

  1. Both the prover and the verifier can sort the combined ttt lexicographically, so that sss can be sorted lexicographically too using typical sorting algorithms
  2. The prover can directly sort sss by ttt, so that the verifier doesn't have to do any pre-sorting and can just rely on the commitment of the columns of ttt (which the prover can evaluate in the protocol).

We take the second approach. However, this must be done carefully since the combined ttt entries can repeat. For some i,li, li,l such that i≠li \neq li=l, we might have

t0[i]+j⋅t1[i]+j2⋅t2[i]=t0[l]+j⋅t1[l]+j2⋅t2[l]t_0[i] + j \cdot t_1[i] + j^2 \cdot t_2[i] = t_0[l] + j \cdot t_1[l] + j^2 \cdot t_2[l]t0​[i]+j⋅t1​[i]+j2⋅t2​[i]=t0​[l]+j⋅t1​[l]+j2⋅t2​[l]

For example, if f={1,2,2,3}f = \{1, 2, 2, 3\}f={1,2,2,3} and t={2,1,2,3}t = \{2, 1, 2, 3\}t={2,1,2,3}, then sorted(f,t)={2,2,2,1,1,2,3,3}\text{sorted}(f, t) = \{2, 2, 2, 1, 1, 2, 3, 3\}sorted(f,t)={2,2,2,1,1,2,3,3} would be a way of correctly sorting the combined vector sss. At the same time sorted(f,t)={2,2,2,2,1,1,3,3}\text{sorted}(f, t) = \{ 2, 2, 2, 2, 1, 1, 3, 3 \}sorted(f,t)={2,2,2,2,1,1,3,3} is incorrect since it does not have a second block of 222s, and thus such an sss is not sorted by ttt.

Recap​

So to recap, to create the sorted polynomials hih_ihi​, the prover:

  1. creates a large query vector which contains the concatenation of the 444 per-row (combined with the joint combinator) queries (that might contain dummy queries) for all rows
  2. creates the (combined with the joint combinator) table vector
  3. sorts all of that into a big vector sss
  4. divides that vector sss into as many hih_ihi​ vectors as a necessary following the snake method
  5. interpolate these hih_ihi​ vectors into hih_ihi​ polynomials
  6. commit to them, and evaluate them as part of the protocol.
