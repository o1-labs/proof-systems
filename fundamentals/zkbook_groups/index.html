Groups

Groups are an algebraic structure which

  • are used in most widely deployed signature schemes

  • are used in most widely deployed zk-SNARKs

  • generalize the notion of a collection of symmetries of an object

  • generalize certain aspects of numbers

I will not touch on the last two too much.

First let's do a quick definition and then some examples.

Definition. A group is a set GGG with a binary operation ∗ ⁣:G×G→G* \colon G \times G \to G∗:G×G→G, an identity e∈Ge \in Ge∈G for that operation, and an inverse map ()−1 ⁣:G→G(\quad)^{-1} \colon G \to G()−1:G→G such that

  • ∗*∗ is associative: x∗(y∗z)=(x∗y)∗zx*(y*z) = (x*y)*zx∗(y∗z)=(x∗y)∗z for all x,y,zx, y,zx,y,z.

  • eee is an identity for ∗*∗. I.e., e∗x=x∗e=xe * x = x * e = xe∗x=x∗e=x for all xxx.

  • ()−1(\quad)^{-1}()−1 is an inverse map for ∗*∗ with identity eee. I.e., x∗x−1=x−1∗x=ex * x^{-1} = x^{-1} * x = ex∗x−1=x−1∗x=e for all xxx.

So basically, an invertible binary operation. Definition in hand, we can see some examples:

  • The integers Z\mathbb{Z}Z with +++, 000 as identity, and negation as inverse.

  • For any natural number nnn, the integers mod nnn. This can be thought of as the set of numbers {0,…,n−1}\{0, \dots, n - 1\}{0,…,n−1} with the operation being +++ followed by taking remainder mod nnn. It can also be thought of as the group of rotations of an nnn-gon.

  • For any set XXX, the set of invertible functions X→XX \to XX→X (a.k.a permutations) with function composition as the operation, the identity function as the identity, and function inverse as the inverse operation.

  • The set of translations of a plane R×R\mathbb{R} \times \mathbb{R}R×R with composition of translations (since the composition of translations is a translation). The identity is translating by (0,0)(0, 0)(0,0) and inverse is reversing the translation. This is equivalent to the group R×R\mathbb{R} \times \mathbb{R}R×R with coordinate-wise addition as the operation and coordinate-wise negation as the inverse.

  • The set of rotations of a sphere with composition as the operation, doing nothing as the identity, and performing the rotation in reverse as the inverse map.

  • For any field FFF, the set F×:=F∖{0}F^{\times} := F \setminus \{0\}F×:=F∖{0}, with field multiplication as the operation, 111 as the identity, and x↦1/xx \mapsto 1 / xx↦1/x as the inverse map. This is called the group of units of FFF.

Sometimes, instead of x∗yx * yx∗y we simply write xyxyxy when ∗*∗ is clear from context.

Abelian groups​

An abelian group is a group in which ∗*∗ is commutative, meaning x∗y=y∗xx * y = y * xx∗y=y∗x. Typically, in that case, we write the group operation as +++ instead and we write the identity as 000.

Some examples are

  • the integers with addition

  • the integers mod nnn with addition

  • the group of units of a field FFF

  • the real numbers with addition

  • the rational numbers with addition

  • the real numbers with multiplication

  • the rational numbers with multiplication

  • vectors of integers of some fixed dimension with coordinate-wise addition

  • The set of polynomials over a ring RRR, R[x]R[x]R[x] with addition

Abelian groups are equivalently described as what are called Z\ZZ-modules. A Z\ZZ-module is like a vector space, but with the integers instead of a field.

Basically a Z\mathbb{Z}Z-module (or equivalently an Abelian group) is a structure where you can add elements and where it makes sense to multiply a group element by an integer.

If GGG is an abelian group, we can define this "multiplication by an integer" as follows. If n≥0n \geq 0n≥0, then for g∈Gg \in Gg∈G, we can define n⋅gn \cdot gn⋅g by

n⋅g:=g+⋯+g⏟nn \cdot g := \underbrace{g + \dots + g}_{n}n⋅g:=ng+⋯+g​​

and if n<0n < 0n<0, then define n⋅g:=∣n∣⋅(−g)n \cdot g := |n| \cdot (-g)n⋅g:=∣n∣⋅(−g). Or equivalently.,

n⋅g=(−g)+⋯+(−g)⏟∣n∣=−(g+⋯+g⏟∣n∣)n \cdot g = \underbrace{(-g) + \dots + (-g)}_{|n|} = -\left( \underbrace{g + \dots + g}_{|n|} \right)n⋅g=∣n∣(−g)+⋯+(−g)​​=−​∣n∣g+⋯+g​​​

This is the general sense of what is called scalar-multiplication or sometimes exponentiation in cryptography.

Cyclic groups​

A cyclic group GGG is a special kind of abelian group. It is an abelian group generated by a single element g∈Gg \in Gg∈G. That is, a cyclic group GGG (generated by g∈Gg \in Gg∈G) is one in which for every h∈Gh \in Gh∈G we have h=ngh = n gh=ng for some n∈Zn \in \mathbb{Z}n∈Z.

Groups in cryptography​

Many cryptographic protocols are defined generically with respect to any abelian group that has a "hard discrete log".

Let

  • GGG be a cyclic group

  • ggg a generator of GGG

  • μ\muμ a probability distribution on GGG

  • A\mathcal{A}A a set of algorithms of type G→ZG \to \mathbb{Z}G→Z with runtime and memory usage bounds. In other words, a set of tuples (P,t,m)(P, t, m)(P,t,m) where PPP is a program of type G→ZG \to \ZG→Z, ttt is a bound on the time that program can be run, and mmm is a bound on the amount of memory that program can use.

    In practice you fix this to be something like, the set of all computations that can be run for less than 1 trillion $.

  • ε∈[0,1]\varepsilon \in [0, 1]ε∈[0,1] a probability, usually taken to be something close to 0 like 1/21281 / 2^{128}1/2128

Then we can define the (G,g,μ,A,ε)(G, g, \mu, \mathcal{A}, \varepsilon)(G,g,μ,A,ε)-computational discrete-log assumption which says:

For any (P,t,m)∈A(P, t, m) \in \mathcal{A}(P,t,m)∈A, if you sample h from G according to μ\muμ, then the probability that P(h)⋅h=gP(h) \cdot h = gP(h)⋅h=g is smaller than ε\varepsilonε, assuming PPP successfully runs within the given resource bounds.

Sometimes this is called the computational Diffie--Helman assumption. Basically what it's saying is that for a group element hhh sampled randomly, you can't practically compute how many times you have to add ggg to itself to get hhh.

Another really important assumption is the no-relation assumption (TODO: name).

Basically what this says is that randomly selected group elements have no efficiently computable linear relations between them. Formally, letting GGG be a group and μ\muμ a probability distribution on GGG, and A\mathcal{A}A a set of programs (with resource bounds) that take in a list of group elements as inputs and outputs a list of integers of the same length.

Then the (G,μ,A,N,ε)(G, \mu, \mathcal{A}, N,\varepsilon)(G,μ,A,N,ε)-no-relation assumption says for all (P,t,m)∈A(P,t,m) \in \mathcal{A}(P,t,m)∈A, for any n≤Nn \leq Nn≤N, if you sample g1,…,gng_1, \dots, g_ng1​,…,gn​ according to μ\muμ, letting [e1,…,en]=P([g1,…,gn])[e_1, \dots, e_n] = P([g_1, \dots, g_n])[e1​,…,en​]=P([g1​,…,gn​]), it is not the case that

e1⋅g1+⋯+en⋅gn=0e_1 \cdot g_1 + \dots + e_n \cdot g_n = 0e1​⋅g1​+⋯+en​⋅gn​=0

except with probability ε\varepsilonε (assuming program PPP runs in time ttt with memory mmm).

Generic group model​

Elliptic curves​

Now, what are some concrete groups that we can safely make the no-relation or computational discrete log hardness assumptions about?

Well, the most efficiently implementable groups that people have come up with -- and that we believe satisfy the above assumptions for A\mathcal{A}A being the class of realistic computations and ε\varepsilonε being something like 1/21281/2^{128}1/2128 -- are elliptic curves over finite fields.

Giving a complete definition of what an elliptic curve is requires a lot of math, and is not very useful from the point of view of cryptography. So we will give a definition that is not complete, but more useful.

An elliptic curve EEE over a field FFF is a set of the form

{(x,y)∈F×F∣y2=x3+ax+b}\{ (x, y) \in F \times F \mid y^2 = x^3 + ax + b \}{(x,y)∈F×F∣y2=x3+ax+b}

for some a,b∈Fa, b \in Fa,b∈F, plus an additional point OOO which is called the point at infinity. Basically it's a set of pairs satisfying some equation of that form. The data of the elliptic curve itself is just the field FFF together with the constants aaa and bbb.

What is special about elliptic curves is that there is a way of giving them a group structure with simple to compute operations that we believe satisfy the hardness assumptions described above.

Group negation (−) ⁣:E→E(-) \colon E \to E(−):E→E is defined by

−(x0,y0)=(x0,−y0)-(x_0, y_0) = (x_0, - y_0)−(x0​,y0​)=(x0​,−y0​)

so we just negate the yyy coordinate.

The identity for the group is OOO, the point at infinity. For that point we may also therefore write it as 000.

Group addition is more complicated to define, so we will not, but here is what's worth knowing about how to compute (x0,y0)+(x1,y1)(x_0, y_0) + (x_1, y_1)(x0​,y0​)+(x1​,y1​)

  • There are three cases

    1. x0≠x1x_0 \neq x_1x0​=x1​

    2. x0=x1x_0 = x_1x0​=x1​ and y0=y1y_0 = y_1y0​=y1​

    3. x0=x1x_0 = x_1x0​=x1​ but y0≠y1y_0 \neq y_1y0​=y1​. In this case it turns out y0=−y1y_0 = -y_1y0​=−y1​ and so the two points are inverse, and we return OOO

    In cases 1 and 2, the algorithm to compute the result just performs a simple sequence of some field operations (multiplications, additions, divisions, subtractions) with the input values. In other words, there is a simple arithmetic formula for computing the result.

Serializing curve points​

Given a curve point (x,y)(x,y)(x,y) we know y2=x3+ax+by^2 = x^3 + ax + by2=x3+ax+b and thus yyy is one of the two square roots of x3+ax+bx^3 + ax + bx3+ax+b. If yyy is a given square root, the other square root is −y-y−y since y2=(−y)2y^2 = (-y)^2y2=(−y)2. So, if we know xxx, then we almost know the whole curve point: we just need a single bit to tell us which of the two possible values (i.e., the two square roots of x3+ax+bx^3+ax+bx3+ax+b) is the y coordinate.

Here is how this is commonly done over prime order fields Fp\mathbb{F}_pFp​, assuming p≠2p \neq 2p=2. Remember that we represent elements of Fp\mathbb{F}_pFp​ as integers in the range [0,p−1][0, p-1][0,p−1]. In this representation, for a field element yyy, −y-y−y is the integer p−yp - yp−y (or 000 if y=0y = 0y=0). Thus, if yyy is odd, then −y-y−y is even (since ppp is odd and an odd minus an odd is even). Similarly, if yyy is even, then −y- y−y is odd (since an odd minus an even is odd).

So, for any yyy, unless yyy is 0, yyy and −y-y−y have opposite parities. Parity can easily be computed: it's just the least significant bit. Thus, we have an easy way of encoding a curve point (x,y)(x, y)(x,y). Send

  • xxx in its entirety

  • The least significant bit b0b_0b0​ of yyy

Given this, we can reconstruct yyy as follows.

  1. Compute any square root YYY of x3+ax+bx^3 + ax + bx3+ax+b

  2. If the least significant bit of YYY is equal to b0b_0b0​, then y=Yy = Yy=Y, otherwise, y=−Yy = -Yy=−Y

In the case of Mina, our field elements require 255 bits to store. Thus, we can encode a curve point in 255+1=256255 + 1 = 256255+1=256 bits, or 32 bytes. At the moment this optimized encoding is not implemented for messages on the network. It would be a welcome improvement.

Projective form / projective coordinates​

The above form of representing curve points as pairs -- which is called affine form -- is sometimes sub-optimal from an efficiency perspective. There are several other forms in which curve points can be represented, which are called projective forms.

The simple projective form represents a curve EEE as above as the set

{(X,Y,Z)∈F3∣(Y/Z)2=(X/Z)3+a(X/Z)+b}\{ (X, Y, Z) \in F^3 \mid (Y/Z)^2 = (X/Z)^3 + a(X/Z) + b \}{(X,Y,Z)∈F3∣(Y/Z)2=(X/Z)3+a(X/Z)+b}

If you think about it, this is saying that (X/Z,Y/Z)(X/Z, Y/Z)(X/Z,Y/Z) is a point on the original curve in affine form. In other words, in projective form we let the first two coordinates get scaled by some arbitrary scaling factor ZZZ, but we keep track of it as the third coordinate.

To be clear, this means curve points have many different representations. If (x,y,z)(x, y, z)(x,y,z) is a curve point in projective coordinates, and sss is any element of FFF, then (sx,sy,sz)(sx,sy,sz)(sx,sy,sz) is another representation of the same curve point.

This means curve points require more space to store, but it makes the group operation much more efficient to compute, as we can avoid having to do any field divisions, which are expensive.

Jacobian form / Jacobian coordinates​

There is another form, which is also sometimes called a projective form, which is known as the jacobian form. There, a curve EEE would be represented as

{(X,Y,Z)∈F3∣(X/Z2,Y/Z3)∈E}\{ (X, Y, Z) \in F^3 \mid (X / Z^2, Y / Z^3) \in E \}{(X,Y,Z)∈F3∣(X/Z2,Y/Z3)∈E}

so the triple (X,Y,Z)(X, Y, Z)(X,Y,Z) corresponds to the affine point (X/Z2,Y/Z3)(X/Z^2, Y/Z^3)(X/Z2,Y/Z3). These are the fastest for computing group addition on a normal computer.

Take aways​

  • Use affine coordinates when the cost of division doesn't matter and saving space is important. Specific contexts to keep in mind:

    • Working with elliptic curve points inside a SNARK circuit

  • Use Jacobian coordinates for computations on normal computers, or other circumstances where space usage is not that costly and division is expensive.

  • Check out this website for the formulas for implementing the group operation.

More things to know​

  • When cryptographers talk about "groups", they usually mean a "computational group", which is a group equipped with efficient algorithms for computing the group operation and inversion. This is different because a group in the mathematical sense need not have its operations be computable at all.
